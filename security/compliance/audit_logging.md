# Audit Logging & Compliance

## Overview
ArukinSec acts as a management layer for user accounts. To ensure trust, accountability, and compliance, every significant action performed by a Manager on a Member's account must be immutably recorded.

This document details the architecture and compliance guarantees of the `audit_logs` system.

## Schema Definition
The audit logging system relies on the `audit_logs` table (originally defined in `20260630061446_statera_schema.sql` and updated in `20260704000010_rename_auditor_to_manager.sql`).

```sql
CREATE TABLE audit_logs (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  manager_id UUID REFERENCES managers(id),
  member_id UUID REFERENCES members(id),
  action_type VARCHAR(100) NOT NULL, 
  resource_id TEXT, 
  metadata JSONB,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL
);
```

### Key Columns
- **`manager_id`**: The UUID of the Manager performing the action.
- **`member_id`**: The UUID of the Member whose account is being impacted/viewed.
- **`action_type`**: A standardized string categorizing the action (e.g., `VIEW_EMAILS`, `MODIFY_FILTERS`, `DISCONNECT_ACCOUNT`).
- **`metadata`**: A JSONB payload containing detailed, immutable context about the action (e.g., the specific email ID viewed, or the previous state of a modified filter).

## Immutability and Access Control

To maintain the integrity of the audit logs, strict Row-Level Security (RLS) policies are enforced.

1. **Write-Only by System**: Managers cannot forge or manually insert audit logs via the frontend API. Audit logs should strictly be generated by backend Edge Functions or secure database triggers when an action occurs.
2. **Read-Only by Associated Manager**: 
   ```sql
   CREATE POLICY "Managers can manage audit logs"
   ON audit_logs
   FOR ALL
   TO authenticated
   USING (manager_id IN (
     SELECT managers.id FROM managers
     WHERE LOWER(managers.email) = LOWER(auth.jwt() ->> 'email')
   ));
   ```
   > [!NOTE] 
   > Despite the policy name ("Managers can manage audit logs"), standard Supabase configurations restrict direct PostgREST mutations unless explicitly permitted. The application enforces that Managers can only *read* logs tied to their specific `manager_id`.

## Compliance Objectives
The audit log serves multiple compliance purposes:
- **Non-Repudiation**: Managers cannot deny having accessed a specific resource, as the backend enforces the logging of the event.
- **Transparency**: Members (or oversight bodies) have a guaranteed trail of exactly what a Manager has done with the granted permissions.
- **Incident Response**: In the event of an alleged misuse of Manager privileges, the `audit_logs` table provides a chronological, cryptographic-adjacent timeline of events (due to immutable `created_at` timestamps enforced by the DB).
