# Audit Logging — Current State

## Status: NOT IMPLEMENTED

The `audit_logs` table existed in early schema versions but was **dropped** in migration `20260704202405_remote_schema.sql`. No replacement audit mechanism currently exists.

### What was planned

The original schema defined an `audit_logs` table with:
- `manager_id` — who performed the action
- `member_id` — which member was affected
- `action_type` — e.g., `VIEW_EMAILS`, `DISCONNECT_ACCOUNT`
- `metadata` — JSONB context about the action
- `created_at` — immutable timestamp

RLS policies restricted reads to the owning manager. Writes were intended to be trigger- or Edge-Function-only (no client-side insert).

### What exists today

- `scan_executions` tracks scan operations (status, category, depth) but not general admin actions (profile views, email reads, setting changes).
- No table records consent grants, privacy policy acceptance, or data access events.
- No audit trail is generated by any Edge Function or database trigger.

### Implications

- **Compliance**: Fails DPDP Act and GDPR requirements for verifiable audit trails of personal data access.
- **Incident response**: No way to reconstruct who accessed what data after a breach.
- **Non-repudiation**: Managers can deny having accessed specific resources.

### Recommended implementation

Re-create an audit table and instrument the Edge Functions:

```sql
CREATE TABLE audit_logs (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  manager_id UUID REFERENCES managers(id),
  member_id UUID REFERENCES members(id),
  action_type VARCHAR(100) NOT NULL,
  resource_id TEXT,
  metadata JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Then instrument key actions in `google-proxy`, `audit-gateway`, `intel-gateway`, and the frontend API calls to write audit records.
